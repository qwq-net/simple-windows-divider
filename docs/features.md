# 機能

windows-divider は次の 3 つの機能を提供します。設定での名前づけに合わせて、それぞれ機能 A・B・C と呼びます。

## 介入の原則（いつウィンドウを動かすか）

windows-divider が能動的にウィンドウを動かすのは、次の 2 つの場面だけです。これ以外でウィンドウを動かすことはありません。

1. 矢印ホットキーを押したとき（機能 B・ユーザーの明示操作）。
2. 新規ウィンドウが生成され、学習した配置に一致したとき（機能 C・自動復元）。

既存ウィンドウのドラッグ・フォーカス移動・表示・最小化解除では動かしません。自動復元は `auto_restore`（トレイの「覚えた配置を自動復元」）で丸ごと無効にできます。

ユーザーが意図しないタイミングでウィンドウが動くこと、これまで戻っていた配置が戻らなくなることは、大きなストレスになります。機能を追加・変更するときも、この 2 つの契機を増やさないこと、既存の学習を無効化しないことを設計上の不変条件とします。

## 機能 A：標準スナップの無効化

Windows 標準のスナップ（Aero Snap / Snap Assist）を、起動時に無効化し、終了時に元へ戻します。

- レジストリ（HKCU）の現在値を退避してから 0 に書き換え、`SystemParametersInfoW(SPI_SETWINARRANGING)` で実行中のシステムにも反映します。
- 退避した内容はファイル（`snap_backup.toml`）にも保存します。これにより、異常終了した場合でも次回起動時に元の設定へ自動で戻します。
- `enabled` が false の間、または `disable_snap` が false の間は、退避していた設定を復元し、スナップには介入しません。

設定は `[general]` の `disable_snap` / `disable_snap_assist` で切り替えます（[configuration.md](configuration.md)）。

## 機能 B：矢印ホットキーによるグリッド配置

各ウィンドウは `columns × rows` のグリッド上で、連続したセル範囲（占有範囲）を持ちます。ホットキーは矢印 4 方向だけで、占有範囲を 1 セルずつ動かします。四隅や全画面は、矢印の組み合わせと同時押しで到達します。

| 操作 | 既定キー | 挙動 |
|---|---|---|
| ← / → | `Ctrl+Alt+Left` / `Right` | 列方向。幅 2 列以上は手前の辺を削って寄せ、幅 1 列はその方向へ 1 列広げる。 |
| ↑ / ↓ | `Ctrl+Alt+Up` / `Down` | 行方向。横の占有は保ったまま同様に動かす。 |
| ← と → を同時 | 反対方向を同時押し | 横軸フル（行は維持）。例：`□□□/■□□` → `□□□/■■■`。 |
| ↑ と ↓ を同時 | 反対方向を同時押し | 縦軸フル（列は維持）。横軸フルと縦軸フルの両方で全画面。 |

左右キーの例（3 列・■ が占有）：全幅 →`←`→ `■■□` →`←`→ `■□□` →`→`→ `■■□` →`→`→ `□■□`。

- 四隅は矢印の組み合わせで到達します（例：`←` で左端へ、続けて `↑` で上端へ＝左上セル）。
- 全画面は、小さい状態から `←`＋`→`（横軸フル）→ `↑`＋`↓`（縦軸フル）の 2 ステップで到達します。専用キーはありません。
- 分割数は `[grid]` の `columns` / `rows` で変えられます（既定はウルトラワイド向けの 3 列 × 2 行）。`auto_aspect` を有効にすると、各ウィンドウが今いるモニタの解像度アスペクト比から分割数を自動判定します（おおよそ 32:9 → 4×2、21:9 → 3×2、16:9 など → 2×2、縦長 → 1×2）。モニタごとに最適な分割になり、`columns`/`rows` は使われません。トレイの「アスペクト比で自動分割」でも切り替えられます。
- 同時押しは待ち時間ゼロで判定します。矢印を押した瞬間に反対方向キーが押されているかを `GetAsyncKeyState` で確認するため、単独押しに遅延は乗りません。
- 最大化中のウィンドウに矢印を押すと、いったん通常状態へ戻し、グリッド全体を占有した状態を起点にします。
- 端でこれ以上動けないとき（その方向の端セルにいて、軸が最小幅の状態）、操作方向に別のモニタがあればそのモニタへ移動します。着地は反対側の端セル（→ なら隣の左端、← なら右端、↑ なら下端、↓ なら上端）で、もう一方の軸の占有は保ちます。例（左右に 2 枚）：モニタ A の右端セルで `→` を押すと、モニタ B の左端セルへ移ります。幅が 2 以上のときは同一モニタ内で縮むだけで移動しません。操作方向にモニタが無ければ従来どおり何も起きません。

配置は、見えるウィンドウの縁がセルにぴったり合うようにします。Windows 10/11 のウィンドウは外周に見えない余白（DWM のリサイズ枠・影領域）を持つため、その分だけ広げて配置し、隣のウィンドウや画面端との隙間が出ないようにしています（`DWMWA_EXTENDED_FRAME_BOUNDS` で実寸を測定）。

占有範囲の起点は次のように決めます。直前に自分が配置した位置（保存済みの占有範囲とウィンドウの現在矩形がほぼ一致する）ならそれを再利用し、手動で動かされていた場合は現在位置から最も近い格子を推定します。

## 機能 C：配置の学習による自動復元

矢印キーでウィンドウを配置したとき、その配置を「スロット」として学習します。同じアプリの新規ウィンドウが現れたら、空いているスロットへ自動で戻します。ルールの手書きは不要です。

- 学習のキーは「実行ファイル名（basename）・ウィンドウクラス名・AppUserModelID（AUMID）」の組です。タイトルは使いません。AUMID を含めることで、同じ実行ファイルとクラスを共有する通常ウィンドウと PWA（インストール済み Web アプリ）を別物として学習できます。
- 記録の単位はスロットで、「どのディスプレイか・そのモニタ内のグリッド占有範囲・記録時の分割数」の組です。1 つのアプリが複数のモニタや位置にスロットを持てます。座標そのものではないため、復元時は記録したディスプレイの作業領域と分割数で矩形へ変換します。分割数が学習時から変わっていても、範囲外のインデックスは丸めて適用します。記録したディスプレイがつながっていなければ、そのスロットへは復元しません。1 つの識別子に貯めるスロットは最大 8 個で、超えると最も古いものから捨てます。
- 学習するのは、矢印キーで配置したときだけです。新規ウィンドウの生成や、既存ウィンドウのドラッグ・フォーカス移動では学習しません。
- 新規ウィンドウが現れたとき、その識別子に記録があり、まだどの窓も使っていない空きスロットがあれば、そのスロットへ配置します。空きスロットはディスプレイ番号と位置の順に詰めます。記録が無いアプリや、スロットがすべて埋まっているとき（同じアプリの 2 つ目以降、タブのドラッグで生まれた窓など）は動かしません。
- 配置済みのウィンドウをドラッグやリサイズで所属スロットから動かすと、そのスロットの所属と記録を外します。次回からはそこへ戻しません。
- 監視には `SetWinEventHook`（out-of-context）を使い、ウィンドウの生成と、ユーザーによる移動・リサイズの完了だけを捉えます。自分の配置（`SetWindowPos`）では移動・リサイズの完了イベントが出ないため、ユーザーの操作だけを所属解除の判定に使えます。
- 生成直後はサイズが確定していないことがあるため、150 ミリ秒の遅延後に最大 3 回まで再配置を試み、目標位置との差が 4 ピクセル以内に収まれば終了します（Electron 系などのリトライ対策）。
- 記録が空のあいだ、または `auto_restore` が false のあいだは、この機能は何もしません。

学習データは設定とは別ファイル（`layouts.toml`）に保存します。連続操作中はメモリ上だけ更新し、最後の操作から約 0.5 秒後にまとめて書き出します（終了時には保留分を必ず書き出します）。自動復元の有効・無効は、設定の `auto_restore` またはトレイの「覚えた配置を自動復元」で切り替えます（[configuration.md](configuration.md)）。

## 動作要件

- Windows 11。スナップ無効化は Windows 11 を対象に調整しています。
- 管理者権限は不要です。非昇格で動かしてください。昇格ウィンドウは UIPI により操作できないため、その種のウィンドウには触れません。

## 既知の制約

- UWP / ストアアプリは、実体が `ApplicationFrameHost.exe` 配下のため best-effort です（うまく動かない場合があります）。
- 昇格ウィンドウは、非昇格プロセスからは UIPI で操作できないため触れません（`SetWindowPos` の ACCESS_DENIED はログに残して継続します）。
- 自動復元は、記録したスロットのディスプレイへ配置します（新規ウィンドウが現れたモニタとは限りません）。記録したディスプレイがつながっていなければ復元しません。
- 同じアプリの複数の窓は、別々のスロットへ配置できます（例：Chrome / Brave の 2 つの窓を別々の区画へ）。記録したスロットがすべて埋まると、それ以上の窓は動かしません。AUMID を取得できないウィンドウは識別子の AUMID が空になるため、同じ実行ファイルとクラスのものはまとめて 1 つの識別子として扱われます。
- タイトルバーもリサイズ枠も持たないウィンドウ（ボーダーレス全画面・オーバーレイ等）は、既定では配置の対象外です。ゲームを避けるための判定で、`skip_non_tileable` で切り替えられます（[anti-cheat.md](anti-cheat.md) / [configuration.md](configuration.md)）。
